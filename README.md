# Делай это, чтобы ускорить FastAPI в 10 раз

Демо-стенд и набор бенчмарков к докладу. Здесь можно своими руками воспроизвести все замеры:
поднять базу, запустить приложение в разных вариантах и обстрелять его `wrk`, а заодно померить
цену ORM, DI, Sentry и флагов uvicorn по отдельности.

Все бенчмарки делятся на две группы:

1. **Клиника** (`bench_clinic/`) — одно и то же реалистичное приложение в шести вариантах.
   Сквозной сценарий «как в проде»: HTTP → DI → сервисы → база. Меряем `wrk`-ом в RPS.
2. **Точечные бенчмарки** (`bench_orm/` и три файла в корне) — изоляция одной технологии.
   Отвечают на вопрос «сколько стоит конкретно вот это»: ORM, `Depends()`, Sentry, флаги uvicorn.

## Требования

| Что                              | Зачем                      | Проверка                 |
| -------------------------------- | -------------------------- | ------------------------ |
| Python 3.14                      | весь код рассчитан на 3.14 | `python -V`              |
| [uv](https://docs.astral.sh/uv/) | менеджер зависимостей      | `uv --version`           |
| Docker + Docker Compose          | Postgres и мок Sentry      | `docker compose version` |
| `wrk`                            | генератор HTTP-нагрузки    | `wrk --version`          |

Установка зависимостей:

```bash
uv sync
```

Создаст `.venv/` со всеми пакетами. Дальше везде используется `.venv/bin/python` и `.venv/bin/uvicorn`,
чтобы не зависеть от того, активировано окружение или нет.

`wrk` ставится отдельно (его нет в pip):

```bash
sudo apt install wrk        # Debian/Ubuntu
brew install wrk            # macOS
```

> **Важно:** все команды запускаются **из корня репозитория**. Приложения и бенчмарки лежат в
> пакетах `bench_clinic` / `bench_orm` и импортируются по полному пути.

## Быстрый старт

```bash
# 1. Поднять Postgres и мок Sentry
docker compose up -d

# 2. Создать и наполнить базу clinic
.venv/bin/python seed_clinic.py

# 3. Запустить приложение клиники
.venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg:app --host 127.0.0.1 --port 8000

# 4. В другом терминале — обстрелять
wrk "http://127.0.0.1:8000/surgery-offer?patient_id=1&procedure_code=rhinoplasty&preferred_date=$(date -d '+7 days' +%Y-%m-%d)&promo_code=NOSIK15" \
  -H "X-User-Id: 1" --connections 64 --duration 15s --threads 1
```

## Секция 1. Приложение клиники (`bench_clinic/`)

### Что за приложение

Клиника пластической хирургии «Силиконовая долина». Ручка `GET /surgery-offer` готовит предварительное
предложение на операцию: проверяет права сотрудника, пациента и его анализы, ищет противопоказания,
подбирает свободный слот с хирургом нужной специализации и операционной, считает смету со скидками.

Ручка **только читает** — ничего не бронирует и не меняет, поэтому её можно обстреливать сколько
угодно, а результат остаётся детерминированным. Happy-path выполняет ровно **10 SQL-запросов**
в одной транзакции — это и есть «полезная работа», на фоне которой мы меряем накладные расходы
фреймворка.

### Демо-сценарии

База засеяна так, чтобы каждый ответ можно было получить предсказуемо. Первые четыре пациента —
заготовленные сценарии (дата ниже — `сегодня + 7 дней`):

| Код   | Сценарий                    | Запрос                                  |
| ----- | --------------------------- | --------------------------------------- |
| `200` | всё хорошо, три скидки      | `patient_id=1`, `promo_code=NOSIK15`    |
| `422` | анализы просрочены/не сданы | `patient_id=2`                          |
| `409` | нет одобренной консультации | `patient_id=3`                          |
| `409` | противопоказание (курение)  | `patient_id=4`                          |
| `403` | недостаточно прав           | тот же запрос с `X-User-Id: 3` (хирург) |
| `401` | неизвестный пользователь    | `X-User-Id: 999`                        |
| `400` | промокод исчерпан / истёк   | `promo_code=KRASOTKA50` / `BOTOX2025`   |

Заголовок `X-User-Id: 1` — администратор, имеет право смотреть предложения.

### Шесть вариантов: лестница оптимизаций

Все шесть файлов реализуют **одну и ту же ручку с одинаковой бизнес-логикой** и возвращают
побайтово идентичный ответ. Отличается ровно один слой — тот, чью цену мы измеряем. Сравнивая
RPS соседних ступеней, получаем вклад каждой оптимизации по отдельности.

| #   | Файл                                    | Что меняется                                                             | Что измеряет (сравнить с)                                                                                   |
| --- | --------------------------------------- | ------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| 1   | `app_clinic_bench_sync.py`              | SQLAlchemy ORM, провайдеры-зависимости — обычные `def`                   | Отправная точка. Синхронные зависимости FastAPI уводит в threadpool (anyio) — измеряемый антипаттерн        |
| 2   | `app_clinic_bench_async.py`             | то же, но все провайдеры — `async def`                                   | Цена sync-зависимостей: разница с №1 при том, что отличаются только ключевые слова                          |
| 3   | `app_clinic_bench_asyncpg.py`           | слой данных — сырой asyncpg вместо SQLAlchemy                            | Цена ORM внутри веб-приложения (сравнить с №2). Сервисы не тронуты: репозитории отдают объекты той же формы |
| 4   | `app_clinic_bench_asyncpg_no_di.py`     | тот же код, но хендлер собирает зависимости **вручную**, без `Depends()` | Чистая цена DI-резолвера FastAPI (сравнить с №3). Тела провайдеров идентичны                                |
| 5   | `app_clinic_bench_asyncpg_starlette.py` | веб-слой — голый Starlette вместо FastAPI                                | Цена слоя FastAPI поверх Starlette (сравнить с №4): валидация, роутинг, сериализация ответа                 |
| 6   | `app_clinic_bench_asyncpg_asgi.py`      | веб-слой — чистая ASGI-функция `app(scope, receive, send)`               | Цена самого Starlette (сравнить с №5): `Request`/`Response`/`Router`/middleware                             |

Начиная с варианта №4 бизнес-логика, репозитории и pydantic-модели полностью совпадают —
ответ по-прежнему сериализуется pydantic-ом. Меняется только веб-слой.

`bench_clinic/app_clinic_bench_sync.py` дополнительно служит **источником схемы БД**: `seed_clinic.py`
и бенчмарки из `bench_orm/` импортируют SQLAlchemy-модели именно оттуда, поэтому засеянная база
гарантированно совпадает с тем, что ожидают приложения.

## Секция 2. Точечные бенчмарки

Здесь веб-фреймворка либо нет вовсе, либо приложение сведено к пустой ручке — чтобы на результат
не влияло ничего постороннего.

### 2.1. Цена ORM без веб-фреймворка (`bench_orm/`)

Берём **те же самые 10 запросов** happy-path `/surgery-offer`, выбрасываем HTTP, ручку и сервисы —
и гоняем их в одной транзакции напрямую. Получаем «транзакций в секунду» (TPS) для каждого ORM
и для сырого драйвера в качестве эталона. Каждый стек делает это **своим идиоматичным способом**,
как написал бы обычный разработчик, а не через `raw SQL`.

| Файл                     | Что меряет                                                                                                                                           |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| `benchmark_async_orm.py` | **Асинхронные** стеки: `asyncpg` (сырой SQL, эталон), SQLAlchemy async, Tortoise, Piccolo, Django ORM. 64 конкурентных воркера, пул на 64 соединения |
| `benchmark_sync_orm.py`  | **Синхронные** стеки: `psycopg2` (сырой SQL, эталон), SQLAlchemy sync, Django ORM, Peewee. 64 обычных потока, у каждого своё соединение              |

Вспомогательные модули — модели и сценарий из 10 запросов для конкретного ORM, сами по себе
не запускаются:

| Файл                   | ORM                                                          | Драйвер  |
| ---------------------- | ------------------------------------------------------------ | -------- |
| `scenario_tortoise.py` | Tortoise ORM                                                 | asyncpg  |
| `scenario_piccolo.py`  | Piccolo                                                      | asyncpg  |
| `scenario_django.py`   | Django ORM (код синхронный, используется в обоих бенчмарках) | psycopg3 |
| `scenario_peewee.py`   | Peewee                                                       | psycopg2 |

Полезные детали:

- **Sanity-check.** Перед замером каждый драйвер прогоняет сценарий один раз, и бенчмарк сверяет
  результаты между собой. Если хоть один вернул не то — запуск падает. Это защита от «быстрого,
  но неправильного» ORM.
- **10 логических запросов, 11 фактических у ORM.** У `asyncpg`/`psycopg2`/SQLAlchemy список
  `required_tests` подтягивается тем же запросом (`joinedload`), а Tortoise/Piccolo/Django/Peewee
  тянут его отдельным. В сводке есть колонка `SQL/txn` — сравнение остаётся честным.
- **Pony ORM не участвует**: последняя версия поддерживает Python только до 3.12.
- **GIL в синхронной версии.** 64 потока — корректная модель: `psycopg2`/`psycopg3` отпускают GIL
  на время сетевых вызовов libpq, поэтому потоки реально перекрываются на ожидании базы.

### 2.2. Отдельные файлы в корне

| Файл                      | Что меряет                                                                                                                                                                                                                  | Ручки                                                             |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| `app_ok.py`               | **Флаги uvicorn.** Минимальное FastAPI-приложение без базы и lifespan: одна ручка, отдающая `OK`. Меряем чисто HTTP/серверный слой — эффект `--http httptools`, `--loop uvloop`, отключённых логов и заголовков             | `GET /ok`                                                         |
| `app_deps_benchmark.py`   | **Цена `Depends()`.** 21 ручка: пустая, плюс по 10 штук с 1…10 синхронными и 1…10 асинхронными зависимостями. Обстреливая их по очереди, видно, как растёт стоимость с числом зависимостей и чем sync-провайдеры хуже async | `GET /empty`, `GET /sync-deps-{1..10}`, `GET /async-deps-{1..10}` |
| `app_sentry_benchmark.py` | **Цена Sentry.** Та же пустая ручка плюс ручка, кидающая исключение. Сравниваем RPS без Sentry, с Sentry и с разным `sample_rate`                                                                                           | `GET /ok`, `GET /error`                                           |
| `sentry_mock_server.py`   | **Не бенчмарк, а заглушка.** Принимает envelope-ы Sentry на порту 9000 и сразу отвечает `200`. Нужен, чтобы мерить оверхед SDK, не завися от сети и реального Sentry. Поднимается автоматически через `docker compose`      | —                                                                 |

```bash
.venv/bin/python seed_clinic.py
```

## Запуск: шаг 3 — приложение (uvicorn)

Общий вид команды:

```bash
.venv/bin/uvicorn <модуль>:app --host 127.0.0.1 --port 8000 [флаги]
```

Доступные цели:

```bash
# Приложения клиники (нужна засеянная база)
.venv/bin/uvicorn bench_clinic.app_clinic_bench_sync:app               --port 8000
.venv/bin/uvicorn bench_clinic.app_clinic_bench_async:app              --port 8000
.venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg:app            --port 8000
.venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg_no_di:app      --port 8000
.venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg_starlette:app  --port 8000
.venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg_asgi:app       --port 8000

# Точечные приложения (база не нужна)
.venv/bin/uvicorn app_ok:app               --port 8000
.venv/bin/uvicorn app_deps_benchmark:app   --port 8000
.venv/bin/uvicorn app_sentry_benchmark:app --port 8000
```

### Флаги uvicorn

| Флаг                  | По умолчанию   | Что делает                                                                             |
| --------------------- | -------------- | -------------------------------------------------------------------------------------- |
| `--http httptools`    | `auto`         | HTTP-парсер на C вместо чистого Python `h11`. Заметно быстрее                          |
| `--loop uvloop`       | `auto`         | Event loop на libuv вместо стандартного asyncio                                        |
| `--no-access-log`     | лог включён    | Убирает запись строчки лога на каждый запрос                                           |
| `--log-level warning` | `info`         | Меньше служебного вывода                                                               |
| `--no-server-header`  | заголовок есть | Не слать `Server:` в каждом ответе                                                     |
| `--no-date-header`    | заголовок есть | Не слать `Date:`                                                                       |
| `--no-proxy-headers`  | включено       | Не разбирать `X-Forwarded-*`                                                           |
| `--workers N`         | 1              | Несколько процессов. Для замеров оверхеда фреймворка держите `1`, иначе меряете железо |

> `auto` у `--http`/`--loop` уже выбирает `httptools` и `uvloop`, если они установлены (а они
> установлены — идут в составе `uvicorn[standard]`). Указывать явно стоит для наглядности и чтобы
> случайная переустановка зависимостей не поменяла результат замера.

### Sentry: включение, отключение и `sample_rate`

Sentry во всех приложениях подключается **через переменные окружения** — в коде ничего править
не нужно:

| Переменная           | Смысл                                                                   |
| -------------------- | ----------------------------------------------------------------------- |
| `SENTRY_DSN`         | Адрес приёмника. **Если не задана — Sentry не инициализируется вообще** |
| `SENTRY_SAMPLE_RATE` | Доля отправляемых событий об ошибках, `0.0`…`1.0`. По умолчанию `1.0`   |

DSN для локального мока: `http://mocked_key@127.0.0.1:9000/1`.

### Примеры запуска

**1. Стандартный запуск, без Sentry** — базовая линия:

```bash
.venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg:app --host 127.0.0.1 --port 8000
```

**2. Стандартный запуск, с Sentry** (`sample_rate` по умолчанию `1.0`):

```bash
SENTRY_DSN=http://mocked_key@127.0.0.1:9000/1 \
  .venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg:app --host 127.0.0.1 --port 8000
```

**3. С Sentry и пониженным `sample_rate`** — сколько экономит семплирование:

```bash
SENTRY_DSN=http://mocked_key@127.0.0.1:9000/1 SENTRY_SAMPLE_RATE=0.1 \
  .venv/bin/uvicorn app_sentry_benchmark:app --host 127.0.0.1 --port 8000
```

**4. Продвинутые флаги uvicorn** — то, ради чего половина доклада:

```bash
.venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg:app \
  --host 127.0.0.1 --port 8000 \
  --http httptools \
  --loop uvloop \
  --no-access-log \
  --no-server-header \
  --no-date-header \
  --no-proxy-headers \
  --log-level warning
```

**5. Всё сразу: быстрые флаги + Sentry + другой пул соединений:**

```bash
SENTRY_DSN=http://mocked_key@127.0.0.1:9000/1 SENTRY_SAMPLE_RATE=0.25 DB_POOL_SIZE=32 \
  .venv/bin/uvicorn bench_clinic.app_clinic_bench_asyncpg:app \
    --host 127.0.0.1 --port 8000 \
    --http httptools --loop uvloop --no-access-log --log-level warning
```

Перед обстрелом убедитесь, что приложение отвечает:

```bash
curl -H 'X-User-Id: 1' \
  "http://127.0.0.1:8000/surgery-offer?patient_id=1&procedure_code=rhinoplasty&preferred_date=$(date -d '+7 days' +%Y-%m-%d)&promo_code=NOSIK15"
```

## Запуск: шаг 4 — обстрел wrk

`wrk` запускается **в отдельном терминале**, пока приложение работает.

### Параметры

| Параметр              | Что означает                                                                                                                                            |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `-c`, `--connections` | Сколько одновременно открытых соединений держать. Это и есть уровень конкурентности. `64` — рабочее значение, совпадает с размером пула БД              |
| `-d`, `--duration`    | Длительность обстрела. `15s` достаточно, чтобы сгладить прогрев; короче `10s` брать не стоит                                                            |
| `-t`, `--threads`     | Сколько потоков у самого `wrk`. Для одного worker-процесса приложения хватает `1` — иначе генератор нагрузки начнёт соревноваться с приложением за ядра |
| `-H`                  | Дополнительный заголовок. Для клиники обязателен `X-User-Id`                                                                                            |
| `--latency`           | Печатать подробное распределение задержек (перцентили)                                                                                                  |

Правило: `--connections` не должен превышать `max_connections` Postgres и имеет смысл
согласовывать с `DB_POOL_SIZE` приложения (по умолчанию оба `64`).

### Пример 1: приложение клиники

```bash
DATE=$(date -d '+7 days' +%Y-%m-%d)

wrk "http://127.0.0.1:8000/surgery-offer?patient_id=1&procedure_code=rhinoplasty&preferred_date=$DATE&promo_code=NOSIK15" \
  -H "X-User-Id: 1" \
  --connections 64 \
  --duration 15s \
  --threads 1 \
  --latency
```

### Пример 2: простая ручка (флаги uvicorn)

База не нужна, приложение — `app_ok:app`:

```bash
wrk http://127.0.0.1:8000/ok --connections 64 --duration 15s --threads 1
```

Здесь RPS на порядки выше, чем у клиники: полезной работы нет, меряется только веб-слой.

### Пример 3: цена зависимостей

Поднимите `app_deps_benchmark:app` и обстреляйте ручки по очереди — разница между ними и есть
стоимость `Depends()`:

```bash
wrk http://127.0.0.1:8000/empty          --connections 64 --duration 15s --threads 1
wrk http://127.0.0.1:8000/async-deps-10  --connections 64 --duration 15s --threads 1
wrk http://127.0.0.1:8000/sync-deps-10   --connections 64 --duration 15s --threads 1
```

### Пример 4: цена Sentry

Один и тот же обстрел, три запуска приложения — без `SENTRY_DSN`, с ним и с пониженным
`SENTRY_SAMPLE_RATE`:

```bash
wrk http://127.0.0.1:8000/ok     --connections 64 --duration 15s --threads 1
wrk http://127.0.0.1:8000/error  --connections 64 --duration 15s --threads 1
```

`/error` кидает исключение на каждый запрос — так видно стоимость отправки событий.
В логах мока (`docker compose logs -f sentry-mock`) будут идти строки `envelope received`.

### Как читать вывод

```
Running 15s test @ http://127.0.0.1:8000/surgery-offer?...
  1 threads and 64 connections
  Thread Stats   Avg      Stdev     Max   +/- Stdev
    Latency    69.34ms  109.43ms 658.46ms   91.85%
    Req/Sec     1.69k   288.30     1.98k    91.11%
  7597 requests in 5.03s, 5.42MB read
Requests/sec:   1511.36
```

- **`Requests/sec`** — главная цифра, её и сравниваем между вариантами.
- **`Latency Avg` / `Max`** — если максимум в разы больше среднего, что-то блокирует event loop
  (типичный симптом синхронных зависимостей).
- **`Non-2xx or 3xx responses`** — если такая строка появилась, замер невалиден: приложение
  отдаёт ошибки, а не работает. Проверьте, что база засеяна и параметры запроса верные.

## Запуск ORM-бенчмарков (без веб-сервера)

Ни uvicorn, ни `wrk` здесь не нужны — только засеянная база. Запускаются как модули из корня:

```bash
# Асинхронные ORM: asyncpg, SQLAlchemy, Tortoise, Piccolo, Django
.venv/bin/python -m bench_orm.benchmark_async_orm

# Синхронные ORM: psycopg2, SQLAlchemy, Django, Peewee (64 потока)
.venv/bin/python -m bench_orm.benchmark_sync_orm
```

По умолчанию: прогрев 3 с, окно замера 10 с, 3 прогона на драйвер. Полный запуск занимает
несколько минут.

Полезные варианты:


## Как мерить честно

Несколько правил, без которых числа не значат ничего:

1. **Один worker.** `--workers 1`. Иначе вы меряете количество ядер, а не цену фреймворка.
2. **`wrk` и приложение не должны драться за CPU.** Держите `--threads 1` у `wrk`; в идеале
   запускайте генератор нагрузки на другой машине.
3. **Прогрев.** Первые секунды уходят на разогрев путей кода, наполнение пула и кеши Postgres.
   Берите `--duration 15s` и не смотрите на первый прогон.
4. **Проверяйте код ответа.** Строка `Non-2xx or 3xx responses` в выводе `wrk` означает, что
   приложение отдаёт ошибки — такой замер надо выбросить.
5. **Меняйте одну вещь за раз.** Вся ценность лестницы из шести приложений в том, что между
   соседними ступенями отличается ровно один слой.
6. **Пул соединений — общий потолок.** Если `--connections` заметно больше `DB_POOL_SIZE`,
   вы меряете ожидание в очереди за соединением, а не фреймворк.
7. **Сравнивайте пропорции, а не абсолютные числа.** На другом железе абсолютные RPS будут
   другими, а соотношения между вариантами — устойчивыми.
